1. خلاصه خیلی مهم
V2Dex یک کلاینت پراکسی/تونل چندسکویی است که UI مشترک آن با React Native نوشته شده و برای macOS، Android و در آینده Windows بک‌اند بومی جدا دارد. هدف اصلی پروژه مدیریت پروفایل‌های V2Ray/Xray/sing-box، انتخاب نود، اتصال/قطع تونل، حالت full tunnel و per-app routing، تست latency، نمایش IP خروجی، وارد کردن subscription/config از clipboard/URI/QR و بسته‌بندی دسکتاپ است.

وضعیت فعلی:
- UI مشترک در src/ فعال است و از App.tsx وارد DashboardScreen می‌شود.
- macOS مسیر عملیاتی اصلی دسکتاپ است: bridge بومی Swift به SingboxRuntime وصل می‌شود، sing-box را اجرا می‌کند، روی 127.0.0.1:2080 mixed proxy می‌سازد، و بسته به حالت، system proxy macOS یا relaunch برنامه‌ها با proxy را مدیریت می‌کند.
- Android مسیر VPN واقعی‌تری دارد: React Native bridge در Kotlin، V2DexVpnService، Xray، TProxy/tun2socks و Android VpnService استفاده می‌شود. local SOCKS روی 43080 است.
- Windows فعلاً بیشتر اسکلت/placeholder است: فایل‌های C# برای bridge/runtime وجود دارد اما RN Windows host کامل و اتصال production کامل نشده است.
- Network Extension macOS در native/V2DexNetworkExtension و قالب‌های entitlement وجود دارد اما طبق README هنوز مسیر signed TUN/VPN production کامل نیست.

SingboxRuntime:
- singleton است.
- binary path را از این مسیرها پیدا می‌کند:
  - explicit path
  - env V2DEX_SINGBOX_PATH
  - app bundle resource sing-box
  - repo .local/bin/sing-box
  - /opt/homebrew/bin/sing-box
  - /usr/local/bin/sing-box
- config را در temp/v2dex-runtime/sing-box-UUID.json می‌نویسد.
- اگر config شامل tun inbound باشد، sing-box را با osascript و administrator privileges اجرا می‌کند و PID را نگه می‌دارد.
- اگر tun نباشد Process عادی اجرا می‌کند.
- منتظر می‌ماند local proxy روی 127.0.0.1:2080 با nc باز شود.
- full mode بدون elevated TUN:
  - MacSystemProxyController.enableProxy برای همه network services فعال می‌شود.
  - AppProxyLauncher.relaunchRunningEnvironmentProxyApps برای اپ‌هایی مثل Telegram که env proxy می‌گیرند انجام می‌شود.
  - backend = system-proxy
- per-app mode:
  - system proxy disable می‌شود.
  - AppProxyLauncher.relaunchSelectedApps فقط appهای enabled و پشتیبانی‌شده را با proxy arguments/env relaunch می‌کند.
  - backend = app-proxy
- stopIfNeeded:
  - sing-box process یا elevated PID را terminate/kill می‌کند.
  - proxyهای macOS را restore/disable می‌کند.
  - appهای relaunch شده را بدون proxy relaunch می‌کند.

MacSystemProxyController:
- از /usr/sbin/networksetup استفاده می‌کند.
- قبل از تغییر proxy، وضعیت web/secure web/socks را برای هر network service ذخیره می‌کند.
- enableProxy برای web, secure web, socks host/port را set و state را on می‌کند.
- disableProxy اگر state ذخیره شده داشته باشد restore می‌کند، وگرنه forceDisableAllProxies انجام می‌دهد.

AppProxyLauncher:
- appهای انتخابی را terminate و با proxy دوباره launch می‌کند.
- برای Chromium family از --proxy-server=http://host:port و flags اضافه استفاده می‌کند.
- برای Telegram و برخی environmentProxyBundleIDs از env vars مثل HTTP_PROXY/HTTPS_PROXY/ALL_PROXY/SOCKS_PROXY استفاده می‌کند.
- هنگام قطع اتصال appها را بدون proxy relaunch می‌کند.

ApplicationDiscovery:
- از NSWorkspace.shared.runningApplications appهای در حال اجرا را می‌گیرد.
- bundleId/name/processName تولید می‌کند.
- حداکثر پیش‌فرض 200.

17. جریان کامل اتصال در macOS از UI تا شبکه
1. کاربر profile/node را انتخاب می‌کند.
2. در UI دکمه connect زده می‌شود.
3. DashboardScreen اکشن connect از useAppStore را صدا می‌زند.
4. appStore.connect اعتبارسنجی می‌کند.
5. startNativeTunnel در src/services/nativeBackend.ts صدا زده می‌شود.
6. چون Platform.OS macos است، buildSingboxConfig کانفیگ sing-box می‌سازد.
7. bridge.startTunnel(configJson, mode, appRulesJson) صدا زده می‌شود.
8. macos/V2DexBridge/V2DexBridge.swift متد startTunnel را می‌گیرد.
9. JSON appRules decode می‌شود و SingboxRuntime.shared.start اجرا می‌شود.
10. SingboxRuntime:
   - sing-box binary را پیدا می‌کند.
   - config را temp می‌نویسد.
   - اگر TUN لازم باشد با administrator privileges اجرا می‌کند.
   - وگرنه Process عادی run می‌کند.
   - تا باز شدن 127.0.0.1:2080 صبر می‌کند.
   - full mode: system proxy macOS را روی 127.0.0.1:2080 فعال می‌کند.
   - per-app: system proxy را disable و appهای selected را با proxy relaunch می‌کند.
11. TunnelStatusSnapshot به JS برمی‌گردد.
12. appStore tunnel state را update می‌کند.
13. refreshTunnelStatus، refreshPing و refreshExitIpInfo اجرا می‌شوند.

18. جریان کامل اتصال در Android از UI تا شبکه
1. کاربر connect می‌زند.
2. appStore.connect -> startNativeTunnel.
3. چون Platform.OS android است، buildXrayConfig کانفیگ Xray می‌سازد.
4. bridge.startTunnel(configJson, mode, appRulesJson) صدا زده می‌شود.
5. V2DexBridgeModule.startTunnel:
   - VpnService.prepare را چک می‌کند.
   - اگر permission لازم باشد Activity permission را باز می‌کند.
   - بعد startVpn را اجرا می‌کند.
6. startVpn، V2DexVpnService را با ACTION_START راه می‌اندازد.
7. V2DexVpnService.startXray:
   - binary و assets را آماده می‌کند.
   - Xray را با config اجرا می‌کند.
   - VPN interface را establish می‌کند.
   - tun fd را به TProxy/tun2socks می‌دهد.
   - traffic VPN به SOCKS 127.0.0.1:43080 و سپس به Xray outbound می‌رود.
8. statusMap به JS وصل می‌شود و UI وضعیت را نشان می‌دهد.

19. جریان import subscription/config
ورودی‌ها:
- Clipboard
- URI/Text field
- QR camera/gallery در Android
- Manual profile برای http/https/socks5

JS flow:
- importFromClipboard یا importFromUri در appStore
- nativeBackend.importProfileFromUri
- native bridge importFromUri اگر موجود باشد
- fallback fetch در JS برای HTTP
- fallback importProfileFromRaw برای raw config

Usage/traffic:
- از header subscription-userinfo خوانده می‌شود.
- fields: upload, download, total, expire
- usedBytes = upload + download
- remainingBytes = total - used
- expiresAt از expire epoch seconds
- گاهی remaining traffic از نام node هم parse می‌شود.

20. تفاوت‌های مهم پلتفرم‌ها
macOS:
- local proxy: 127.0.0.1:2080
- runtime: sing-box
- full mode: system proxy یا در صورت TUN elevated sing-box
- per-app: process rules در sing-box و/یا relaunch app با proxy args/env
- persistence: UserDefaults
- QR unavailable

Android:
- local proxy: 0.0.0.0/127.0.0.1:43080
- runtime: Xray + VpnService + tun2socks/TProxy
- full mode: VPN همه traffic به جز خود app
- per-app: addAllowedApplication برای packages selected
- persistence: SharedPreferences
- QR camera/gallery موجود

Windows:
- planned: sing-box + Wintun
- فعلاً placeholder و غیرproduction

21. محدودیت‌ها و ریسک‌های فعلی
- پروژه git repo نیست یا در این مسیر .git ندارد؛ git status خطا می‌دهد.
- README می‌گوید scaffold/starter foundation است، production-ready signed VPN app نیست.
- macOS Network Extension production wiring کامل نیست.
- Windows native backend کامل نیست.
- Android import Kotlin فقط VLESS را parse می‌کند؛ اما TS/Swift پشتیبانی بیشتری دارند.
- JS configParser فقط vless/hysteria2/tuic را پشتیبانی می‌کند، اما singboxBuilder outboundهای بیشتری دارد.
- دو منبع config builder وجود دارد: JS و Swift. باید مراقب drift بین src/services/singboxBuilder.ts و native/V2DexCore/Sources/SingboxConfigBuilder.swift بود.
- برای macOS full mode system proxy روی همه network services تغییر می‌کند؛ cleanup/restore مهم است.
- per-app macOS با relaunch برنامه‌ها کار می‌کند و ممکن است برای همه appها پشتیبانی نشود.
- TUN macOS elevated با osascript admin privilege است، نه Network Extension production.
- Android فقط arm64-v8a binaryها در مسیر فعلی دیده می‌شود؛ ABIهای دیگر باید بررسی/اضافه شوند اگر release عمومی لازم است.
- DNS leak protection در مدل state هست اما رفتار دقیق آن بسته به runtime/config فعلی باید جداگانه audit شود.

6. cleanup macOS proxy در crash/termination app تقویت شود.

8. خطاها و stateهای اتصال در UI دقیق‌تر تفکیک شوند: binary missing، permission denied، proxy port timeout، subscription parse failure.
